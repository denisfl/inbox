## Context

The user wants Inbox to be the single daily attention center. Currently only user-initiated content enters the system. News aggregation adds a passive, automatic input channel: configured RSS feeds are fetched overnight, and a concise digest is assembled and delivered via Telegram each evening.

The tech stack already has everything needed: SolidQueue with recurring jobs (`config/recurring.yml`), Ollama for summarization, Telegram bot for delivery, and the Document model for storage.

## Goals / Non-Goals

**Goals:**

- Fetch news from user-configured RSS/Atom feed URLs on a nightly schedule
- Deduplicate articles by URL so re-fetched feeds don't create duplicates
- Store each article as a Document (`source: 'news'`) tagged with the feed name
- Assemble and send an evening digest via Telegram summarizing the day's news
- Summary generated by Ollama (best-effort; fallback to item list if Ollama fails)
- News sources configured in a YAML file (no DB admin UI needed for v1)

**Non-Goals:**

- Full-text web scraping beyond RSS description fields
- Per-feed read/unread tracking
- User-facing news feed UI (Telegram digest is the primary delivery; documents are browsable as a fallback)
- Push notifications / real-time updates

## Decisions

### Decision: RSS/Atom via `feedjira` gem

**Chosen:** `feedjira` — battle-tested Ruby RSS/Atom parser, handles edge cases, minimal setup.

**Alternative:** `nokogiri` raw HTML scraping — more flexible but fragile. Rejected for v1; can be added as a separate story for non-RSS sources.

### Decision: Sources in `config/news_sources.yml`

```yaml
sources:
  - name: "Хабр"
    url: "https://habr.com/ru/rss/best/daily/"
    tag: "tech"
  - name: "The Verge"
    url: "https://www.theverge.com/rss/index.xml"
    tag: "tech"
```

Simple, version-controlled, no DB migration needed.

### Decision: Deduplication by URL stored on Document

Add `url` column to `documents` table. `FetchNewsJob` checks `Document.exists?(url: item.url)` before creating. Requires migration.

### Decision: Evening digest via `SendNewsDigestJob` at 20:00

- Queries all documents with `source: 'news'` created today
- Groups by feed tag
- Calls Ollama to generate a 3–5 sentence summary per group (best-effort; fallback to bullet list)
- Sends formatted Telegram message to `TELEGRAM_ALLOWED_USER_ID`

### Decision: SolidQueue recurring schedule

```yaml
production:
  fetch_news:
    class: FetchNewsJob
    schedule: at 2am every day
  send_news_digest:
    class: SendNewsDigestJob
    schedule: at 8pm every day
```

### Decision: Ollama summarization (best-effort)

Same `gemma3:4b` model. Prompt: "Summarize these news items in 3–5 sentences in Russian. Be concise and factual." If Ollama fails, send raw item titles as a bullet list.

## Risks / Trade-offs

- **Risk:** RSS feed URLs change or go offline → **Mitigation:** Job rescues per-feed errors; other feeds continue; failed feeds logged.
- **Risk:** Many articles per night → large document count → **Mitigation:** Configurable `max_items_per_feed` in sources YAML (default: 20).
- **Risk:** `url` column migration required → **Mitigation:** Simple `add_column :documents, :url, :string`; index on `url` for fast deduplication.
- **Risk:** Ollama slow/unavailable at 20:00 → **Mitigation:** 120s timeout; fallback to plain title list.

## Migration Plan

1. Add `url` string column to `documents` + index
2. Add `gem 'feedjira'` to Gemfile
3. Create `config/news_sources.yml`
4. Implement `FetchNewsJob`
5. Implement `SendNewsDigestJob`
6. Update `config/recurring.yml` with two new schedules
7. Deploy; restart worker container
